# Pitfalls Research

**Domain:** AI-powered Vietnamese Tu Vi astrology interpretation web app
**Researched:** 2026-03-23
**Confidence:** HIGH (scraper/calendar/AI hallucination), MEDIUM (scoring port, tone enforcement), HIGH (streaming infra)

---

## Critical Pitfalls

### Pitfall 1: cohoc.net Scraper Breaks Because Results Are Client-Side Rendered

**What goes wrong:**
A `requests` + BeautifulSoup scraper against `tuvi.cohoc.net/lap-la-so-tu-vi.html` returns an empty `.chart` div or incomplete HTML every time. The page is a JavaScript-driven single-page app — the la so is generated by JavaScript after form submission and rendered into the DOM client-side. A static HTTP fetch never executes that JavaScript, so the chart data never appears.

**Why it happens:**
The cohoc.net page uses JavaScript variables (`g_namSinh`, `g_thangCur`, `g_gioSinh`, `g_isNam`) and a token-based security string (`g_token_key`) to validate and calculate the chart entirely in-browser. There is no server endpoint that returns the la so as a plain HTML page or JSON payload that can be called directly without reverse-engineering the token system.

**How to avoid:**
Use Playwright (Python) to drive a headless Chromium browser: fill the form fields programmatically, click the "lấy lá số" button, wait for `waitForSelector('.chart')` or `waitForLoadState('networkidle')` to confirm the chart has rendered, then parse the resulting DOM. This is the only reliable scraping approach for JS-rendered forms. Build from day one with Playwright, not `requests`.

**Warning signs:**
- Scraper returns an HTML page with an empty or `min-height: 400px` placeholder `.chart` div
- BeautifulSoup finds no sao names (Tử Vi, Thiên Cơ, etc.) in the parsed HTML
- Response is identical regardless of birth data inputs

**Phase to address:**
Data pipeline phase (scraper build). Do not proceed to scoring engine until the Playwright scraper is verified to return complete sao placements for at least 5 different birth inputs.

### Pitfall 2: cohoc.net HTML Structure Changes Silently and Breaks Parsing

**What goes wrong:**
The scraper runs correctly for weeks, then a site update changes a CSS class name, restructures the chart table, or renames data attributes. The scraper continues to run without errors — it just returns empty or wrong sao data. Downstream scoring produces incorrect dimension scores, and AI interpretations are generated on garbage input. Users and the expert partner may not notice for several days.

**Why it happens:**
The scraper relies on specific CSS selectors or element positions in the DOM. A single front-end change to cohoc.net invalidates these selectors silently. There is no contract or API — the site can change at any time without notice. Production scrapers that only log HTTP errors (not semantic validation errors) miss this failure mode entirely.

**How to avoid:**
After each scraper parse, validate the extracted data structure against a schema before passing it downstream:
- Exactly 12 cung must be present
- Each cung must contain at least 1 sao (chinh tinh or phu tinh)
- Known chinh tinh (Tử Vi, Thiên Cơ, Thái Dương, etc.) must appear somewhere in the chart
- Numeric positions must be in range 1-12

If validation fails, raise a `ScraperStructureError` and serve a graceful error to the user rather than proceeding with malformed data. Log the raw HTML for diagnosis.

**Warning signs:**
- Sao count per cung drops to 0 or 1 across all inputs
- Chinh tinh names are absent from extracted data
- Scores are all at the baseline value (no variance across users)
- Expert partner flags that interpretations are generic/identical across readings

**Phase to address:**
Data pipeline phase. Implement schema validation as part of the initial scraper build, not as a future enhancement.

### Pitfall 3: Vietnamese Lunar Calendar Intercalary Month Mis-classification

**What goes wrong:**
A user born during an intercalary (leap) lunar month is assigned to the wrong month. For example, if a user was born in intercalary month 3 (thang 3 nhuan), the conversion code may treat it as regular month 3 or month 4, producing an entirely wrong la so. The error is invisible to the user but produces a chart for the wrong birth date.

**Why it happens:**
The Vietnamese lunisolar calendar inserts a leap month approximately every 3 years. The rule for determining which month becomes the leap month is astronomically derived (the first lunar month after Winter Solstice that contains no Principal Solar Term). Most naive Gregorian-to-Am-lich libraries either ignore leap months entirely or handle them incorrectly. Additionally, Vietnamese and Chinese calendars diverge in specific years (documented: 2007, 2030, 2053) and Vietnam uses GMT+7 (not GMT+8 like China), meaning a new moon at 23:00 UTC+7 starts a new Vietnamese month that China starts a day later.

**How to avoid:**
Use Ho Ngoc Duc's well-documented Vietnamese lunar calendar algorithm (xemamlich.uhm.vn) as the authoritative reference, not a generic Chinese calendar library. In the input form, present the intercalary month option explicitly: if a user selects lunar calendar input, show a checkbox "Tháng nhuận" (leap month) alongside the month selector. Store the raw input flag and pass it through to the scraper. Test with known edge-case birth dates: intercalary months in 1985 (leap month 2), 1995 (leap month 8), 2001 (leap month 4), 2023 (leap month 2).

**Warning signs:**
- Users who know their own chart flag that the sao placements are wrong
- Two users with adjacent birth dates (one in normal month 3, one in intercalary month 3) receive identical charts
- The expert partner finds the chart outputs for certain birth dates "impossible" per traditional rules

**Phase to address:**
Data pipeline phase — Am lich conversion step. Validate the converter against a reference table of known Gregorian-to-Am-lich conversions before integrating with the scraper.

### Pitfall 4: AI Generates Interpretations Not Grounded in Provided Chart Data

**What goes wrong:**
Claude produces luan giai narrative that references star combinations, alerts, or life themes not present in the user's actual la so. The AI "fills in" plausible-sounding Tu Vi content from its general training knowledge rather than staying strictly within the supplied sao placements and scores. Expert partner review catches fabricated references to stars not in the user's chart.

**Why it happens:**
Claude was trained on astrology content and has internalized general Tu Vi patterns. When the prompt context is ambiguous or the knowledge-base injection is loose (e.g., "here is Tu Vi knowledge, now interpret this chart"), the model synthesizes plausible content rather than grounding strictly in the provided data. This is especially likely for dimension-specific interpretations where only a few relevant sao are active — the model will pad the narrative with generic domain content.

**How to avoid:**
Structure the system prompt with an explicit grounding constraint: "Only reference stars, cung, and scores that appear in the data block below. Do not reference any star or combination not listed in `<chart_data>`. If no relevant stars are present for this dimension, state that directly." Pass the extracted sao placements as a structured data block in XML tags, not as prose. After generation, run a post-validation pass: extract star names from the generated text and verify each appears in the chart data. Flag responses containing unrecognized star names for human review.

**Warning signs:**
- AI response mentions a specific sao by name that does not appear in the chart data
- Expert partner flags that the AI described a combination that is not present in this chart
- Different users with very different charts receive similar dimension narratives

**Phase to address:**
AI luan giai engine phase. Build the grounding validation before testing with real users.

### Pitfall 5: AI Uses Fear-Inducing Language Despite Tone Instructions

**What goes wrong:**
Despite system prompt instructions to be "positive and empowering," Claude generates phrases like "cần chú ý vì có thể gặp tai họa nghiêm trọng" (should be careful as serious misfortune may occur) or "nguy cơ bệnh nặng" (risk of serious illness) when negative alerts (e.g., ky sao combinations) are present. Users in the MVP test group are distressed rather than empowered.

**Why it happens:**
Tone instructions in the system prompt are often overridden by the model's domain knowledge when the input data contains genuinely negative indicators. The model "knows" certain sao combinations are inauspicious and defaults to conveying that accurately even when told not to. Single-pass system prompt instructions are insufficient to consistently enforce tone — especially for moderately negative alerts where the boundary between "informative" and "fear-inducing" is contextual.

**How to avoid:**
Implement tone enforcement at three levels: (1) System prompt with explicit prohibition and reformulation rules: "Never state outcomes as inevitable. Never use words: tai họa, nguy hiểm, thất bại, bệnh nặng, chết. For every negative indicator, follow this structure: [current challenge] → [protective action] → [positive outcome pathway]." (2) Include 2-3 few-shot examples in the prompt showing how to handle the worst-case alert categories (e.g., sao indicating health risks) in an empowering way. (3) Post-generation scan for prohibited vocabulary — flag and regenerate if found. Have the expert partner review and sign off on tone exemplars before the MVP launch.

**Warning signs:**
- Expert partner flags responses as "too scary" or "not how I would say this"
- MVP user feedback includes words like "lo lắng" (worried/anxious) in response to the app
- Generated text contains conditional doom patterns: "nếu không cẩn thận sẽ..." (if not careful, will...)

**Phase to address:**
AI luan giai engine phase. Tone exemplar review with expert partner should be a gate before any user testing.

### Pitfall 6: Google Sheets Scoring Logic Ports Incorrectly to Python

**What goes wrong:**
The ported Python scoring engine produces different dimension scores than the validated Google Sheets version for the same birth data. The discrepancy is small (e.g., 1-2 points) and only appears for edge cases, but it causes the expert partner to lose trust in the engine. Worse, some lookups silently return None instead of raising an error, producing 0-scores for dimensions that should have moderate values.

**Why it happens:**
Google Sheets and Python differ in: (1) integer vs. float division (`/` in Python 3 always returns float; Sheets TRUNC behavior differs from Python `int()`), (2) VLOOKUP approximate match semantics vs. Python dict lookups (Sheets may match the nearest lower value while Python raises KeyError on exact-match failure), (3) case sensitivity in string comparisons (sao names may have diacritic inconsistencies), (4) empty cell behavior (Sheets returns 0 for empty cells used in arithmetic; Python raises TypeError or returns None).

**How to avoid:**
Port the scoring logic by treating the Google Sheet as a test oracle. For every input in the existing Sheets validation dataset, the Python engine must produce exactly the same score. Build a test fixture from the expert's existing readings: extract 10-20 known la so + expected scores, run them through the Python engine, and assert exact equality. Use `decimal.Decimal` for score arithmetic to avoid floating-point drift. Normalize all sao name strings (strip whitespace, normalize Unicode diacritics with `unicodedata.normalize('NFC', s)`) before lookups. Raise `KeyError` explicitly on missing lookup keys — never silently return 0 or None.

**Warning signs:**
- Python score differs from Sheets score for the same input, even by 1 point
- Some dimensions always score 0 for unusual birth hour inputs
- Alert detection misses combinations that the expert's Sheets version catches

**Phase to address:**
Scoring engine phase. Parity tests against the Sheets oracle must pass before the engine is used for any real user data.

### Pitfall 7: Streaming Responses Silently Truncated by Vercel/Proxy

**What goes wrong:**
Streaming AI responses work perfectly in local development and in Railway-to-browser direct testing. After deploying to production (Vercel frontend proxying to Railway backend), responses are either: (a) delivered entirely in one burst after a 60-second wait instead of streaming token-by-token, or (b) truncated at an arbitrary character count with no error shown to the user.

**Why it happens:**
Vercel's serverless functions have a 10-second timeout on the Hobby plan (60 seconds on Pro). Next.js API routes buffer the response body until `res.end()` is called — they do not natively stream SSE without specific configuration. Nginx and other reverse proxies buffer streaming responses unless `X-Accel-Buffering: no` is set. The frontend EventSource or ReadableStream implementation may also collapse multiple rapid chunks into one render update.

**How to avoid:**
Route streaming directly from the React frontend to the Railway FastAPI backend — do not proxy through Next.js API routes. Set `X-Accel-Buffering: no` and `Cache-Control: no-cache` headers on the FastAPI streaming endpoint. Send SSE keep-alive comments (`: ping\n\n`) every 15 seconds to prevent proxy connection closure. Verify streaming end-to-end in a staging environment with Vercel + Railway before any user testing, not just in localhost. Test with `curl --no-buffer` to confirm tokens arrive progressively.

**Warning signs:**
- In the browser Network tab, the streaming response shows a single large chunk arriving after a delay rather than progressive small chunks
- Long generations (>30 seconds) appear to complete but then show no output
- SSE `EventSource` `onmessage` fires once with the full payload instead of many times

**Phase to address:**
Streaming UI phase. Verify the streaming pipeline in the deployed staging environment before completing the phase.

### Pitfall 8: Shareable URLs Expose Sensitive Personal Birth Data

**What goes wrong:**
Shareable result URLs encode birth date, time, and gender in the URL path or query parameters (e.g., `/result?d=1985-03-15&h=07&g=female`). A user shares their chart link in a public forum. Third parties can enumerate or guess URLs for other users' birth data. Vietnamese birth hour and date are also sensitive personal identifiers — combined with name, they can be used for social engineering.

**Why it happens:**
The no-auth MVP design optimizes for frictionless sharing. Developers default to encoding all state in the URL because there is no session. Birth data is the natural primary key for the chart. The sensitivity of this data is underestimated — birth date + time + gender is more uniquely identifying than a name in some contexts.

**How to avoid:**
Generate opaque UUIDs as the shareable link key and store the birth data server-side (cache), returning only the UUID in the URL (e.g., `/result/a1b2c3d4`). The UUID is unguessable and does not expose birth data. On the input form, include a clear data handling note: "Thông tin này chỉ dùng để tính lá số, không lưu trữ tên hay thông tin cá nhân." Do not log birth data in application logs.

**Warning signs:**
- URL contains recognizable date/time patterns
- URL is predictable from known birth data
- Two users with the same birth data produce the same URL (hash collision risk, enumerable)

**Phase to address:**
User flow and caching phase. Design the URL scheme before building the result page — retrofitting opaque IDs after URLs are shared is very difficult.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| `requests` + BeautifulSoup scraper | Simpler setup, no headless browser | Never works for JS-rendered cohoc.net; requires full rewrite to Playwright | Never — use Playwright from the start |
| Hardcode cohoc.net selectors without validation | Faster initial build | Silent data corruption when site changes; no detectable failure mode | Never — add schema validation alongside selectors |
| Inline birth data in shareable URLs | No database needed | Privacy exposure; users may share sensitive data publicly | MVP only if using opaque hash (not raw birth params) |
| Skip Am lich intercalary month handling | Simpler input form | Wrong charts for users born in leap months (~1 in 3 years affected) | Never — affects real users |
| Single-pass system prompt for tone control | Less prompt engineering | Expert partner rejects outputs; user distress during testing | Never — tone enforcement needs examples + post-scan |
| Port scoring logic without parity tests | Faster development | Expert loses trust if scores diverge from validated Sheets; hard to audit | Never — tests are the only way to verify correctness |
| Proxy streaming through Next.js API routes | Simpler architecture | Streaming breaks silently in Vercel production due to buffering | Never — connect frontend directly to FastAPI for streams |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| cohoc.net scraper | Using `requests.get()` expecting chart HTML | Use Playwright, fill form fields, wait for `.chart` selector to render, then parse DOM |
| cohoc.net scraper | Waiting for `networkidle` on every scrape | `networkidle` can hang on ad/analytics requests; use `waitForSelector('.chart [data-cung]')` with a timeout |
| cohoc.net scraper | Not handling `g_token_key` | Token may be required for form submission to proceed; inspect network traffic to determine if server validation occurs |
| Claude API streaming | Using `stream=True` with Next.js API routes | Next.js buffers the response body; use SSE directly from FastAPI to frontend React EventSource |
| Claude API structured output | Assuming JSON schema compliance is 100% | Always validate returned JSON against schema server-side; treat parse errors as recoverable with a retry |
| Am lich conversion library | Using a Chinese calendar library without timezone adjustment | Must use GMT+7 (Hanoi time), not GMT+8 (Beijing time); divergence is 1 hour and causes date boundary errors around midnight |
| Am lich conversion | Not providing intercalary month flag to cohoc.net | The cohoc.net form likely has a "tháng nhuận" option; if birth month is intercalary and flag is not set, the chart will be for the wrong month |
| Scoring engine | Treating missing sao as score=0 | Missing lookup key means the sao is not present, not that it scores 0; absent keys should be handled as "not applicable" with a specific sentinel |
| FastAPI on Railway | Default Uvicorn timeout too short for AI generation | Claude Sonnet can take 20-60 seconds for a full dimension response; set `--timeout-keep-alive 120` on Uvicorn |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Generating all 7 dimension luan giai upfront on chart load | 60-180 second page load; timeout errors; high API cost per visit | Use on-demand generation per dimension as specified in PROJECT.md | Day 1 if implemented wrong |
| Running Playwright scraper synchronously per user request (no cache) | Every user waits 5-15 seconds for scraper to launch browser, fill form, wait for render | Cache la so results by hash(birth_date + birth_hour + gender + calendar_type); TTL: indefinite (chart never changes) | At ~5 concurrent users |
| Storing full luan giai text in SQLite as unindexed text blobs | Fast initially; slow when serving cached results at scale | Index cache by input hash; for MVP SQLite is fine up to ~500 charts | At ~500 cached charts (not relevant for MVP test group) |
| Playwright instance per scraper call | Memory exhaustion; slow startup | Reuse a persistent browser context with a pool; close and reopen on error | At ~3 concurrent scraper requests |
| Streaming 7 dimensions simultaneously | Claude API rate limits hit; frontend renders chaotically | Stream one dimension at a time on user click; other dimensions idle | Immediately if implemented naively |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Logging full birth data (date, hour, gender) in application logs | Personal data exposure in log aggregation systems; GDPR-adjacent risk | Log only the opaque cache key (UUID); never log raw birth parameters |
| Encoding birth data in shareable URL parameters | Enumerable; SEO-indexed; exposes user data to anyone with the link | Use opaque UUID keys; map to birth data server-side in cache only |
| Exposing Claude system prompt contents in API responses | Expert's curated knowledge base and prompt engineering leaked to users | Never return system prompt in API responses; use `is_error: true` handling that returns only a user-facing message |
| Injecting unvalidated user input into Claude prompts | Prompt injection: user could enter birth data fields containing instructions to Claude | Treat all user-provided values as data, not instructions; pass them inside XML delimiters (`<user_input>`) in the prompt; validate types (year: integer 1900-2100, etc.) before injection |
| No rate limiting on scraper endpoint | A single user could trigger 100 Playwright browser sessions, exhausting Railway server memory | Rate limit by IP: max 3 chart requests per minute; return 429 with retry-after header |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Showing a blank page or spinner with no progress feedback during 5-15 second scraper + AI generation | Users assume the app is broken; high abandonment rate | Show a multi-step progress indicator: "Đang lập lá số..." → "Đang phân tích sao..." → "Đang viết luan giai..." |
| Displaying raw sao names (e.g., "Tử Vi, Thiên Phủ, Lộc Tồn") without explanation in the UI | Non-expert users (the MVP test group are clients, not practitioners) have no context | Show sao names as secondary metadata; lead with the scored dimension results and narrative |
| Presenting negative alert badges prominently before the empowering narrative | Users fixate on red alert icons before reading the reassuring text | Place alerts after the positive narrative; use neutral or amber color (not red) for alert markers |
| All 7 dimensions visible immediately with no hierarchy | Decision paralysis; users don't know where to start | Surface the top 2-3 dimensions by score first; let users expand others |
| Generic error message when cohoc.net is unavailable ("An error occurred") | Users don't know if they should retry, wait, or contact support | Show: "Không thể lập lá số lúc này. Vui lòng thử lại sau 5 phút." with a retry button; log the failure server-side |
| No confirmation of Am lich vs. Duong lich input before generating chart | Users unsure which calendar they entered get a chart for the wrong system | Show a clear summary before submission: "Ngày sinh: 15/03/1985 (dương lịch)" with an edit button |

## "Looks Done But Isn't" Checklist

- [ ] **Scraper:** Returns valid chart data — verify by checking that all 14 chinh tinh are distributed across 12 cung with no duplicates
- [ ] **Am lich conversion:** Handles intercalary months — test with a birth date known to fall in a leap month (e.g., Gregorian 1985-04-05 which falls in Am lich intercalary month 2, 1985)
- [ ] **Scoring engine:** Python output matches Google Sheets output exactly — run 10 known la so through both and assert equality
- [ ] **AI luan giai:** Only references stars present in the chart — run post-generation validation for each response before serving to user
- [ ] **Tone enforcement:** No fear-inducing language — expert partner must review and sign off on at least 3 generated samples per dimension category before go-live
- [ ] **Streaming:** Works in Vercel + Railway production, not just localhost — test with `curl --no-buffer` against the deployed endpoint
- [ ] **Shareable URLs:** Do not expose birth data — confirm URL contains only opaque ID, not date/time/gender parameters
- [ ] **Cache:** Same birth data returns same chart — submit the same birth data twice and verify the second call hits cache (no scraper run)
- [ ] **Error handling:** cohoc.net unavailability shows graceful error — kill network connectivity to cohoc.net in test and verify the error UI appears
- [ ] **Intercalary month UI:** Form shows "tháng nhuận" option — manually submit a leap month birth date and confirm cohoc.net receives the correct flag

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Scraper built with `requests`; cohoc.net returns empty charts | HIGH | Rewrite scraper with Playwright; retest all validation cases; no salvageable code from `requests` approach |
| Scores diverge from Sheets by >1 point after launch | HIGH | Halt AI generation; fix scoring engine; re-generate any cached luan giai that used incorrect scores; expert re-review |
| cohoc.net changes HTML structure mid-production | MEDIUM | Detect via schema validation; update selectors; cached charts unaffected; only new requests blocked until fix deployed |
| AI generated fear-inducing language reaches users | MEDIUM | Add post-generation vocabulary scan immediately; manually review all cached luan giai in the database; notify expert partner |
| Shareable URLs expose birth data | HIGH | Generate new opaque IDs for all cached charts; invalidate and redirect old URLs; cannot un-share already-distributed links |
| Am lich intercalary month bug discovered post-launch | MEDIUM | Identify affected birth dates; invalidate their cache entries; users must regenerate their charts; no data loss but trust impact |
| Vercel streaming broken in production | LOW | Switch frontend to fetch directly from Railway API (bypass Next.js proxy); deploy fix in <1 hour |
